## Реализация библиотеки

### Конфигурации

#### Неблокирующая MPMC-очередь (алгоритм Michael-Scott)

##### Назначение и решаемые проблемы

Классическая lock-free очередь на базе связного списка, обеспечивающая корректную работу множества продюсеров и консьюмеров (MPMC) без использования мьютексов в критической секции.

- **Решаемые проблемы:** устранение задержек, связанных с переключением контекста в ядре (context switch), исключение инверсии приоритетов и гарантия прогресса системы (lock-freedom).
- **Роль в исследовании:** высокопроизводительный эталон для сценариев с экстремальной нагрузкой (contention), сравнение стоимости динамической аллокации узлов против статического кольцевого буфера.

##### Поведение

Динамическое хранение данных в виде односвязного списка узлов. В отличие от кольцевых буферов, память выделяется под каждое сообщение отдельно. Очередь всегда содержит один «фиктивный» (dummy) узел для разделения головы и хвоста.

- **Механизм:** использование атомарных операций `Compare-and-Swap` (CAS) для вставки в хвост и извлечения из головы.
- **Взаимопомощь (Helping):** если поток обнаруживает, что указатель `tail` отстал от фактического последнего узла, он пытается продвинуть `tail` самостоятельно, прежде чем продолжить свою операцию.
- **Управление памятью:** для предотвращения **ABA-проблемы** и безопасного удаления узлов используются политики Memory Reclamation (Hazard Pointers или Epoch-based).
- **Синхронизация (согласно ADR):** методы `Send` и `Read` остаются блокирующими на уровне интерфейса (используют `condition_variable` для ожидания данных/места), но сама логика вставки/удаления в `StoreMessage` реализована без блокировок.
- **Сложность:** запись / чтение — **O(1)**; память — **O(N)**, где N — количество сообщений.

**race condition риски**
Поскольку алгоритм изначально разработан для MPMC, риски race condition минимизированы на уровне атомарных операций.

| Сценарий | Риск |
| -------- | ---- |
| SPSC/SPMC/MPSC | Безопасно, минимальные накладные расходы |
| MPMC | Основной сценарий: CAS-циклы разрешают конфликты конкурентных потоков |

**Реализация пайплайна (Overriding)**
Согласно общему ADR, логика распределена следующим образом:
- `Prework()`: Проверка флага `Close` и контроль `capacity` (атомарный счетчик). Если очередь полна/пуста — блокировка на `condition_variable`.
- `StoreMessage()`: Lock-free алгоритм Майкла-Скотта (цикл CAS для перестановки указателей `next`, `head`, `tail`).
- `Postwork()`: Уведомление (`notify_one`) ожидающих потоков о появлении нового сообщения или свободного места.

##### Интерфейс

**Параметры конкретного инстанса очереди**
| Параметр | Описание |
| ----------- | -------- |
| MessageType | Тип сообщения |
| capacity | Логический лимит сообщений (для реализации backpressure) |
| Reclamation | Политика очистки памяти (например, `HazardPointers`) |
| Allocator | Аллокатор для создания узлов списка |

**Публичные методы**
|  метод  | поведение         |
| ------- | ----------------- |
| Send    | Продюсер добавляет сообщение. Если достигнут `capacity` — поток блокируется. Вставка в список выполняется без мьютекса. |
| TrySend | Продюсер пробует добавить сообщение. Если очередь полна — ивент `OVERFLOW`. |
| Read    | Консьюмер забирает сообщение. Если очередь пуста — блокируется. Извлечение из списка — lock-free. |
| TryRead | Консьюмер забирает сообщение. Если пусто — ивент `EMPTY`. |
| Size    | Возвращает текущее значение атомарного счетчика элементов. |
| Stats   | Статистика: число успешных операций, количество «промахов» CAS (retries), использование памяти. |
| Close   | Разблокирует все потоки. После закрытия `Read` отдает остатки, `Send` возвращает ошибку. |
