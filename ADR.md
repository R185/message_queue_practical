## Реализация библиотеки

### Базовый класс

#### Методы

| публичный метод | поведение |
| ------ | -------- |
| Send | Продюссер отправляет сообщение. Если буффер заполнен -- блокируется поток продюсера, пока не освободится место |
| TrySend | Продюсер отправляет сообщение, если буфер не заполнен; иначе возвращает `false` |
| Read | Консьюмер читает сообщение, если буфер не пустой. Иначе блокируется поток, пока не появится сообщение. Сообщение на `head` удаляется |
| TryRead | Консьюмер читает сообщение, если буфер не пустой. Иначе возвращает `std::nullopt` |
| Size | Возвращает количество сообщений в очереди |
| Empty | Возвращает `true`, если очередь пуста |
| Clear | Завершает очередь, разблокировывая все потоки |

-------------------

#### Параметры

| шаблонный параметр | описание |
| -------- | -------- |
| `ValueType` | Тип сообщения. **concept `MessageType`**: MoveConstructible и CopyConstructible |
| `ThreadCategory` (`ThreadAccessCategory`) | Категория доступа потоков: `kSingleProducerSingleConsumer`, `kMultipleProducerSingleConsumer`, `kSingleProducerMultipleConsume`, `kMultipleProducerMultipleConsumer` |
| `ExceptionPolicy` (`DeadlockExceptionPolicy`) | Политика обработки дедлоков и смены роли потока (см. ниже) |

| `DeadlockExceptionPolicy` | описание |
| -------- | -------- |
| `kOnThreadRoleChange` | Кидать исключение, если поток сначала был consumer, а затем вызвал `Send` (или наоборот для `Read`) |
| `kOnDeadlock` | Кидать исключение при дедлоке (например, consumer вызывает `Send` при полном буфере в SPSC) |
| `kNoException` | Не кидать исключение; поток помечается как `kBoth` |

> Некоторые шаблонные параметры можно переместить к обычным параметрам инстанса. Добавлены сюда, так как появится возможность использовать `if constexpr`.

| параметры инстанса | описание |
| -------- | -------- |
| `timeout` | Опциональный параметр, при котором очередь заблокирует поток только на определённый срок, после чего вернёт пустой результат или выбросит исключение |

*Остальные параметры должны быть определены у классов-наследников* (например `allocator`, `capacity`).

#### Dynamic polymorphism vs. CRTP

- Dynamic polymorphism
  | + | - |
  | --------- | -------- |
  | Универсальность для любых сегментов памяти (в том числе динамическое владение объектов, например в `std::vector`) | Долгое образение к виртуальным методам |
  | Хорошая композиция | Нету возможности оптимизировать под стек циклические буфферы без аллокатора |

- CRTP
  | + | - |
  | --------- | -------- |
  | Оптимизация до C методов | Не совместим с динамической аллокацией |
  | Отсутствие виртуальных таблиц |  |

**Выбор CRTP** - это упор на скорость.
**Выбор Dynamic polymorphism** - это упор на универсальность.

> Вероятно польза от скорости CRTP не оправдана, так как она будет амартизирована обращением к динамической области памяти буффера (если буффер не выделяется в стеке). Вероятно стоит выбрать Dynamic polymorphism, так как он будет более универсальным и будет работать с любыми сегментами памяти.

#### overriding методов

**Реализовывать overriding методов стоит по такому типу:**
``` cpp
CheckSendDeadlockPossibility() = 0;
SyncAndOverflowPrework() = 0;
TrySyncAndOverflowPrework() = 0;
StoreMessage(const ValueType& message) = 0;
StoreMessage(ValueType&& message) = 0;
SendPostwork() = 0;

void SendPrework() {
  AccountThreads(ThreadRole::kProducer);
  if (CheckSendDeadlockPossibility()) {
    HandleDeadlock();
  }
  SyncAndOverflowPrework();
}

bool TrySendPrework() {
  AccountThreads(ThreadRole::kProducer);
  if (CheckSendDeadlockPossibility()) {
    return false;
  }
  return TrySyncAndOverflowPrework();
}

void Send(ValueType&& message) {
  SendPrework();
  StoreMessage(std::move(message));
  SendPostwork();
}
```

**overriding методы `Read` / `TryRead`:**

``` cpp
void ReadPrework() {
  AccountThreads(ThreadRole::kConsumer);
  if (CheckReadDeadlockPossibility()) {
    HandleDeadlock();
  }
  SyncAndUnderflowPrework();
}

bool TryReadPrework() {
  AccountThreads(ThreadRole::kConsumer);
  if (CheckReadDeadlockPossibility()) {
    return false;
  }
  return TrySyncAndUnderflowPrework();
}

ValueType Read() {
  ReadPrework();
  ValueType message = PopMessage();
  ReadPostwork();
  return message;
}

std::optional<ValueType> TryRead() {
  if (!TryReadPrework()) {
    return std::nullopt;
  }
  ValueType message = PopMessage();
  ReadPostwork();
  return message;
}
```

**Pipeline публичных методов**

``` mermaid
flowchart LR
    A[Send] --> M0(Account threads)
    M0 --> M1(Check send deadlock)
    M1 -->|True| M2(Handle deadlock)
    M1 -->|False| M3(Sync and overflow prework)
    M2 -->|No exception| M3
    M3 --> M4(Store message)
    M4 --> M5(Send postwork)
```

**Pipeline `TrySend`**

``` mermaid
flowchart LR
    A[TrySend] --> M0(Account threads)
    M0 --> M1(Check send deadlock)
    M1 -->|True| M2(Return false)
    M1 -->|False| M3(Try sync and overflow prework)
    M3 -->|False| M2
    M3 -->|True| M4(Store message)
    M4 --> M5(Send postwork)
    M5 --> M6(Return true)
```

**Ответственности методов**

``` mermaid
flowchart LR
    A[IMessageQueue] --> M0(Account threads)
    A --> M2(Handle deadlock)
    A --> M1s(Check send deadlock)
    A --> M1r(Check read deadlock)
    B[Inherited class] --> M3o(Sync and overflow prework)
    B --> M3ot(Try sync and overflow prework)
    B --> M4s(Store message)
    B --> M5s(Send postwork)
    B --> M3u(Sync and underflow prework)
    B --> M3ut(Try sync and underflow prework)
    B --> M4r(Pop message)
    B --> M5r(Read postwork)
```

| overriding метод | ответственность |
| ------------ | --------------- |
| `CheckSendDeadlockPossibility()` | Проверка вырожденного сценария: поток с ролью consumer пытается записать в полную очередь |
| `SyncAndOverflowPrework()` | Синхронизация и резолвинг переполнения для блокирующего `Send` (ожидание на `not_full` или drop oldest) |
| `TrySyncAndOverflowPrework()` | Неблокирующий аналог; возвращает `false`, если запись невозможна |
| `StoreMessage()` | Запись сообщения в буфер (перегрузки для lvalue и rvalue) |
| `SendPostwork()` | Пробуждение ожидающих consumer, уведомление семафоров/Condition Variables |

**Pipeline `Read`**

``` mermaid
flowchart LR
    A[Read] --> M0(Account threads)
    M0 --> M1(Check read deadlock)
    M1 -->|True| M2(Handle deadlock)
    M1 -->|False| M3(Sync and underflow prework)
    M2 -->|No exception| M3
    M3 --> M4(Pop message)
    M4 --> M5(Read postwork)
```

**Pipeline `TryRead`**

``` mermaid
flowchart LR
    A[TryRead] --> M0(Account threads)
    M0 --> M1(Check read deadlock)
    M1 -->|True| M2(Return nullopt)
    M1 -->|False| M3(Try sync and underflow prework)
    M3 -->|False| M2
    M3 -->|True| M4(Pop message)
    M4 --> M5(Read postwork)
    M5 --> M6(Return optional)
```

**overriding методы для `Read` / `TryRead`**

| overriding метод | ответственность |
| ------------ | --------------- |
| `CheckReadDeadlockPossibility()` | Producer вызывает `Read` при пустой очереди (симметрия `CheckSendDeadlockPossibility`) |
| `SyncAndUnderflowPrework()` | Синхронизация и ожидание элемента для блокирующего `Read` |
| `TrySyncAndUnderflowPrework()` | Неблокирующая проверка; `false`, если читать нечего |
| `PopMessage()` | Атомарное извлечение с `head`; при исключении элемент остаётся в очереди |
| `ReadPostwork()` | Пробуждение producer |

### Конфигурации

Эталонная ограниченная FIFO-очередь «писатели → читатели» с естественным backpressure: переполнение тормозит быстрых продюсеров вместо роста памяти или потери данных.

- **Решаемые проблемы:** разграничение скоростей продюсеров и консюмеров при жёстком лимите памяти и гарантии отсутствия потерь.
- **Роль в исследовании:** базовая (reference) стратегия для сравнения с кольцевыми (drop-oldest / block-writer) и lock-free по пропускной способности, латентности, памяти и fairness.

#### Кольцевая очередь с политикой переполнения «блокировка писателя»

##### Поведение (блокировка писателя)

В этой конфигурации происходит one-time аллокация буффера размера `capcity`. 
Сообщения хранятся в циклическом буффере (запись на tail, чтение с head).
**При попытке послать сообщение при переполненом буффере блокируется поток продюсера** (реализуется в `SyncAndOverflowPrework`: ожидание на `not_full`).

При одном чтении, разблокироваться должен 1 продюсер.

При попытке вызвать `Send` в единственном потоке-консьюмере при переполнении буффера приведёт к *deadlock*
Параметризация эксепшенов: 
- Кидать эксепшен при консюмере, который стал продюссером.
- Кидать эксепшен при дедлоке.
- Не кидать.

Значит для политик дедлока нужно сохранять теги у потоков через `thread_local ThreadRole` (`AccountThreads`).

Из-за отсутвия замещения целостность данных гарантируется.

Категория доступа потоков `MC` будет приводить к тому, что каждое сообщение будет отходить только одному потоку. Логично? не очень. Может быть потребуется реализовать `Broadcast` метод.

**race condition риски**
| Категория доступа потоков | риск  |
| ------------------------- | ----- |
| SPSC                      | нет |
| SPMC                      | `Read` |
| MPSC                      | `Send` |
| MPMC                      | `Read` и `Send` |

**Синхронизация**
| Категория доступа потоков | синхронизация |
| ------------------------- | ------------- |
| SPSC                      | Достаточно 2 симофоров: Количество свободных мест и количество занятых мест |
| MPSC                      | mutex на `Send` и `TrySend` + 2 симофоров |
| SPMC                      | mutex на `Read` и `TryRead` + 2 симофоров |
| MPMC                      | mutex на всё + 2 симофоров или condition variables `not_full` и `not_empty` |

*Скорее всего есть синхронизации оптимальнее*

#### Кольцевая очередь с политикой переполнения «сброс самого старого»

Кольцевая очередь на статическом буфере с вытеснением самого старого элемента при заполнении.  
Продюсер никогда не блокируется; консьюмер может ожидать появления элементов.  
Переполнение обрабатывается в `SyncAndOverflowPrework`: вытеснение oldest вместо блокировки.

##### Поведение (сброс самого старого)

###### Опустошение и переполнение

| Аспект                                                 | Решение                                                                                                                                    |
| ------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------ |
| Поведение при заполнении (продюсер)                    | Новый элемент всегда принимается. Если очередь заполнена, самый старый элемент удаляется, освобождая место. Поток-продюсер не блокируется. |
| Поведение при пустой очереди (блокирующий консьюмер)   | Метод `take()` блокирует вызывающий поток до появления элемента. Поток засыпает на условной переменной `notEmpty`.                         |
| Поведение при пустой очереди (неблокирующий консьюмер) | Метод `tryTake()` немедленно возвращает пустое значение (`std::nullopt`), если очередь пуста, не блокируясь.                               |
| Очерёдность пробуждения консьюмеров                    | При вставке элемента, если очередь была пуста, пробуждается **один** ожидающий консьюмер (FIFO).                                           |

###### Потокобезопасность
| Аспект                       | Решение                                                                                                                                                  |
| ---------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Модель конкурентного доступа | **MPMC** – любое количество продюсеров и консьюмеров могут одновременно вызывать любые методы очереди.                                                   |
| Механизм синхронизации       | **Один общий мьютекс** + **одна условная переменная** `notEmpty` для ожидания элементов.                                                                 |
| Атомарность операций         | Каждая операция захватывает мьютекс, выполняет действия с индексами/счётчиком и освобождает мьютекс. Вставка при необходимости вытесняет старый элемент. |
| Справедливость ожидания      | **FIFO** для очереди ожидающих консьюмеров.                                                                                                              |

###### Прерывания
| Аспект                   | Решение                                                                                                                                      |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------- |
| Подверженные методы      | Только блокирующие методы консьюмера: `take()` и `take(timeout)`.                                                                            |
| Механизм отмены ожидания | Метод `take()` и `take(timeout)` принимают токен отмены (`std::stop_token`) или ссылку на атомарный флаг отмены.                             |
| Реакция на сигнал отмены | При поступлении сигнала отмены поток пробуждается, **не извлекает элемент** (очередь не меняется) и выбрасывает исключение `QueueCancelled`. |
| Сообщение об отмене      | Исключение `QueueCancelled` (от `std::runtime_error`). Второй вариант – возврат `std::nullopt`.                                              |

###### Асимптотики

| Операция                    | Время | Память |
| --------------------------- | ----- | ------ |
| `put(T)`                    | O(1)  | O(1)   |
| `take()`                    | O(1)  | O(1)   |
| `tryTake()`                 | O(1)  | O(1)   |
| `peek()`                    | O(1)  | O(1)   |
| `size()`                    | O(1)  | O(1)   |
| `capacity()`                | O(1)  | O(1)   |
| `empty()`                   | O(1)  | O(1)   |
| `full()`                    | O(1)  | O(1)   |
| **Общая память очереди**    | —     | O(N)   |

##### Интерфейс (сброс самого старого)

**Параметры конкретного инстанса очереди**
| параметр    | описание |
| ----------- | -------- |
| MessageType | Тип сообщения |
| capacity    | Максимум сообщений в очереди (размер кольцевого буфера) |

| Метод                                   | Описание                                                                                                           |
| --------------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| `Queue(size_t capacity)`                | Создаёт очередь с фиксированной ёмкостью. `capacity > 0`.                                                          |
| `void put(item)`                        | Добавляет элемент. Если очередь заполнена, вытесняет самый старый элемент. Поток не блокируется.                   |
| `T take(stop_token)`                    | Извлечение элемента с блокировкой.                                                                                 |
| `optional<T> take(timeout, stop_token)` | Извлечение элемента с блокировкой и таймаутом. При успехе (по `timeout`) возвращает элемент, иначе `std::nullopt`. |
| `std::optional<T> tryTake()`            | Извлечение элемента без блокировки. Если очередь пуста, возвращает `std::nullopt`.                                 |
| `std::optional<T> peek()`               | Возвращает копию головного элемента без извлечения. `std::nullopt`, если очередь пуста.                            |
| `size_t size() const`                   | Текущее количество элементов в очереди.                                                                            |
| `size_t capacity() const`               | Максимальная ёмкость очереди.                                                                                      |
| `bool empty() const`                    | `true`, если очередь пуста.                                                                                        |
| `bool full() const`                     | `true`, если очередь полностью заполнена.                                                                          |

| Исключение              | Описание                                        |
| ----------------------- | ----------------------------------------------- |
| `QueueCancelled`        | Бросается `take` при запросе отмены.            |
| `std::invalid_argument` | Бросается в конструкторе, если `capacity == 0`. |

#### Очередь на deque с политикой переполнения «блокировка писателя»

Динамическое хранение в `std::deque` (по умолчанию) или `std::list`; буфер **не преаллоцируется** — память выделяется по мере роста. Отличие от кольцевых конфигураций и предмет отдельного сравнения «аллокации vs преаллоцированное кольцо». FIFO: запись в хвост (`push_back`), чтение с головы (`pop_front`).

##### Поведение (deque)

- При переполнении (`size == capacity`) блокируется поток продюсера до освобождения места; при пустой очереди блокируется поток консьюмера до появления сообщения.
- Одно чтение разблокирует одного продюсера; одна запись — одного консьюмера.
- Замещения и потерь нет → целостность данных гарантируется.
- `Send` из единственного потока-консьюмера при полном буфере → *deadlock* (вырожденный сценарий: единственный потребитель сам себя блокирует).
- Сложность: запись / чтение — амортизированно **O(1)**; память — **O(`capacity`)** плюс накладные контейнера.
- `std::deque` / `std::list` **не потокобезопасны даже в SPSC**: одновременные `push_back` и `pop_front` гонятся за общим состоянием (`size`, узлы, карта блоков), поэтому взаимное исключение требуется во всех сценариях конкуренции.

**race condition риски** (сценарии задаются числом потоков-продюсеров/консюмеров в бенчмарке)
| Сценарий | риск  |
| -------- | ----- |
| SPSC     | `Send` ↔ `Read` (общий контейнер не потокобезопасен) |
| SPMC     | `Send` ↔ `Read` + `Read` ↔ `Read` |
| MPSC     | `Send` ↔ `Read` + `Send` ↔ `Send` |
| MPMC     | `Read` и `Send` со всех сторон |

Где `SPSC` — Single Producer Single Consumer, `SPMC` — Single Producer Multiple Consumers, `MPSC` — Multiple Producers Single Consumer, `MPMC` — Multiple Producers Multiple Consumers.

**Синхронизация**

Во всех сценариях единообразно: один `mutex` вокруг контейнера + две `condition_variable` (`not_full`, `not_empty`); ожидание с проверкой предиката (защита от ложных пробуждений и потерянных уведомлений); `Close` будит всех ожидающих при остановке. *Оптимизация: two-lock (раздельные замки head/tail на связном списке) повышает параллелизм продюсер/консюмер — отложено.*

##### Интерфейс (deque)

**Параметры конкретного инстанса очереди**
| параметр    | описание |
| ----------- | -------- |
| MessageType | Тип сообщения |
| capacity    | Максимум сообщений в очереди |
| container   | Бэкенд хранения: `std::deque` (по умолчанию) или `std::list` |

**Публичные методы**
|  метод  | поведение         |
| ------- | ----------------- |
| Send    | Продюсер отправляет сообщение. Если очередь заполнена — блокируется поток продюсера, пока не освободится место |
| TrySend | Продюсер отправляет сообщение, если очередь не заполнена; иначе возвращает `false` |
| Read    | Консьюмер читает сообщение с головы (с удалением). Если очередь пуста — блокируется поток, пока не появится сообщение |
| TryRead | Консьюмер читает сообщение с головы (с удалением). Если очередь пуста — возвращает `std::nullopt` |
| Size    | Текущее число сообщений в очереди |
| Stats   | Статистика очереди: число записей/чтений, потери (`0` для этой стратегии), время и количество блокировок, пиковый размер |
| Close   | Закрыть очередь: разбудить всех ожидающих; после закрытия `Read` отдаёт остаток и завершает, `Send` отклоняется |

#### Приоритетная очередь на основе min-heap с сохранением FIFO-порядка

Ограниченная приоритетная очередь, гарантирующая обработку сообщений с наивысшим приоритетом (минимальным числовым значением) в первую очередь. Для сообщений с одинаковым приоритетом строго гарантируется порядок FIFO (First-In-First-Out).

- **Решаемые проблемы:** обеспечение QoS (Quality of Service) и своевременная обработка критически важных или срочных сообщений в условиях высокой конкуренции и ограниченных вычислительных ресурсов. Предотвращение голодания (starvation) равноприоритетных сообщений за счет сохранения хронологического порядка их поступления.
- **Роль в исследовании:** целевая стратегия для проверки профиля нагрузки «смесь сообщений с разными приоритетами». Будет сравниваться с O(1) очередями для оценки накладных расходов на поддержание инварианта кучи (логарифмическая сложность) и влияния алгоритма на пропускную способность и латентность.

##### Поведение

###### Опустошение и переполнение

| Аспект                                                 | Решение                                                                                                                                                                                |
| ------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Поведение при заполнении (блокирующий продюсер)        | Если очередь достигла `capacity`, метод вставки блокирует вызывающий поток до тех пор, пока консьюмер не извлечет хотя бы один элемент. Поток засыпает на условной переменной `notFull`. |
| Поведение при пустой очереди (блокирующий консьюмер)   | Метод `take()` блокирует вызывающий поток до появления элемента. Поток засыпает на условной переменной `notEmpty`.                                                                     |
| Поведение при пустой очереди (неблокирующий консьюмер) | Метод `tryTake()` немедленно возвращает `std::nullopt`, если очередь пуста, не блокируясь.                                                                                             |
| Очерёдность извлечения сообщений                       | Всегда извлекается элемент с минимальным значением приоритета (Min-Heap). При равенстве приоритетов извлекается тот, который был добавлен раньше (гарантия FIFO).                      |

###### Потокобезопасность

| Аспект                       | Решение                                                                                                                                                                                          |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Модель конкурентного доступа | **MPMC** – любое количество продюсеров и консьюмеров могут одновременно обращаться к очереди.                                                                                                    |
| Механизм синхронизации       | **Один общий мьютекс** + **две условные переменные** (`notEmpty` для консьюмеров, `notFull` для продюсеров).                                                                                     |
| Атомарность операций         | Каждая операция захватывает мьютекс. Инкремент внутреннего счетчика последовательности (sequence ID) для обеспечения FIFO происходит строго под мьютексом одновременно с операцией `push_heap`.  |
| Справедливость ожидания      | Ожидание потоков на условных переменных делегируется планировщику ОС. Внутренний порядок сообщений строго детерминирован (Priority + FIFO).                                                      |

###### Прерывания

| Аспект                   | Решение                                                                                                                                             |
| ------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| Подверженные методы      | Только блокирующие методы: `take()`, `take(timeout)`, `put()` и `put(timeout)`.                                                                     |
| Механизм отмены ожидания | Методы принимают токен отмены (`std::stop_token`) из стандарта C++20/23.                                                                            |
| Реакция на сигнал отмены | При поступлении сигнала поток пробуждается, **прерывает операцию** (вставка не происходит, элемент не извлекается) и выбрасывает `QueueCancelled`. |
| Сообщение об отмене      | Исключение `QueueCancelled` (наследуется от `std::runtime_error`).                                                                                  |

###### Асимптотики

| Операция                    | Время       | Память |
| --------------------------- | ----------- | ------ |
| `put(T, priority)`          | O(log N)    | O(1)   |
| `take()`                    | O(log N)    | O(1)   |
| `tryTake()`                 | O(log N)    | O(1)   |
| `peek()`                    | O(1)        | O(1)   |
| `size()`                    | O(1)        | O(1)   |
| `capacity()`                | O(1)        | O(1)   |
| `empty()`                   | O(1)        | O(1)   |
| `full()`                    | O(1)        | O(1)   |
| **Общая память очереди**    | —           | O(N)   |

Динамическое хранение на базе непрерывного массива (`std::vector` под управлением алгоритмов `std::push_heap` и `std::pop_heap`). 

- **Механизм стабилизации кучи (FIFO):** Стандартная куча нестабильна. Для сохранения FIFO порядка элементы оборачиваются во внутреннюю структуру `{priority, sequence_id, payload}`. Поле `sequence_id` заполняется из инкрементируемого монотонного счетчика (типа `uint64_t`) в момент захвата мьютекса. Компаратор кучи сначала сравнивает `priority`, а при их равенстве — `sequence_id`.
- При переполнении (`size == capacity`) блокируется поток продюсера; при пустой очереди блокируется поток консьюмера.
- Сложность: запись и чтение требуют перестроения бинарного дерева, что дает логарифмическую сложность **O(log N)** в худшем случае. Пиковая производительность будет ниже, чем у O(1) стратегий (кольцевых очередей).
- Внутренний контейнер не потокобезопасен, перестроение кучи (`push_heap`/`pop_heap`) требует монопольного доступа, поэтому взаимное исключение (`std::mutex`) охватывает всю операцию целиком.

**race condition риски** (сценарии задаются числом потоков-продюсеров/консюмеров в бенчмарке)

| Сценарий | риск  |
| -------- | ----- |
| SPSC     | `Send` ↔ `Read` (модификация вектора и перестроение кучи не атомарны) |
| SPMC     | `Send` ↔ `Read` + `Read` ↔ `Read` (гонка при `pop_heap`) |
| MPSC     | `Send` ↔ `Read` + `Send` ↔ `Send` (гонка за инкремент `sequence_id` и `push_heap`) |
| MPMC     | `Read` и `Send` со всех сторон |

**Синхронизация**

Во всех сценариях применяется единый подход: один `mutex` защищает структуру данных (вектор + счетчик `sequence_id`) + две `condition_variable` (`not_full`, `not_empty`); ожидание происходит с проверкой предикатов `size < capacity` и `size > 0` (защита от ложных пробуждений); метод `Close` будит всех ожидающих при остановке системы.

##### Интерфейс

**Параметры конкретного инстанса очереди**

| параметр    | описание |
| ----------- | -------- |
| MessageType | Базовый тип передаваемого сообщения. |
| PriorityType| Тип приоритета (например, `int` или `enum`). Меньшее значение = высший приоритет. |
| capacity    | Максимальное число сообщений в очереди. |

**Публичные методы**

| Метод                                            | Описание                                                                                                                                                |
| ------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Queue(size_t capacity)`                         | Создаёт приоритетную очередь с фиксированной ёмкостью. Выделяет базовую память, если используется `reserve`.                                            |
| `void put(item, priority, stop_token)`           | Добавляет элемент с указанным приоритетом. Если очередь заполнена, блокируется до появления свободного места или получения сигнала отмены.              |
| `bool tryPut(item, priority)`                    | Отправляет сообщение, если очередь не заполнена (возвращает `true`). Если заполнена — немедленно возвращает `false` (событие `OVERFLOW`).               |
| `T take(stop_token)`                             | Извлекает элемент с наивысшим приоритетом с блокировкой. Если очередь пуста, блокируется до появления элемента.                                         |
| `optional<T> take(timeout, stop_token)`          | Извлечение элемента с блокировкой и таймаутом. При успехе возвращает элемент, по истечении `timeout` возвращает `std::nullopt`.                         |
| `std::optional<T> tryTake()`                     | Извлечение элемента с наивысшим приоритетом без блокировки. Если очередь пуста, немедленно возвращает `std::nullopt` (событие `EMPTY`).                 |
| `std::optional<T> peek()`                        | Возвращает копию головного (наивысший приоритет) элемента без извлечения.                                                                               |
| `size_t size() const`                            | Текущее количество элементов в приоритетной очереди.                                                                                                    |
| `Stats getStats() const`                         | Возвращает агрегированную статистику: пропускная способность, число записей/чтений, время блокировок потоков, пиковый размер очереди.                   |
| `void close()`                                   | Переводит очередь в закрытое состояние: будит все потоки. После закрытия новые `put` отклоняются, `take` отдает остатки кучи, затем бросает исключение. |

| Исключение              | Описание                                                                                                 |
| ----------------------- | -------------------------------------------------------------------------------------------------------- |
| `QueueCancelled`        | Бросается при прерывании ожидания в `take()` или `put()` через `stop_token`.                             |
| `QueueClosed`           | Бросается, если вызывается `put` в закрытую очередь, или `take`, когда закрытая очередь уже опустошена. |
| `std::invalid_argument` | Бросается в конструкторе, если `capacity == 0`.                                                          |
