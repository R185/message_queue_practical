# Политика тестирования библиотеки очередей сообщений

## 1. Общие положения

### 1.1 Цели тестирования
- Подтверждение корректности реализации всех конфигураций очередей.
- Обеспечение потокобезопасности
- Проверка поведения при граничных условиях
- Оценка производительности и выявление узких мест.

### 1.2 Инструменты
- **Фреймворк:** Google Test

### 1.3 Общая структура тестов
Для каждой конфигурации:
1. Функциональные тесты (однопоточные и с контролируемым чередованием).
2. Многопоточные тесты
3. Cancellation тесты для блокирующих методов.
4. Особые случаи

---

## 2. Сценарии тестирования по конфигурациям

### 2.1 Блокирующая очередь на основе списка или дека

**Особенности:** динамическая аллокация узлов, блокировка продюсера при заполнении, блокировка консьюмера при опустошении, FIFO.

#### 2.1.1 Функциональные тесты
| Тест                  | Входные данные / действия                                                  | Ожидаемый результат                                             |
| --------------------- | -------------------------------------------------------------------------- | --------------------------------------------------------------- |
| `PutTakeSingle`       | Вставить один элемент, извлечь.                                            | Извлечённый элемент равен вставленному.                         |
| `FifoOrder`           | Вставить N элементов, затем извлечь N.                                     | Порядок извлечения совпадает с порядком вставки.                |
| `BlockingPutOnFull`   | Заполнить очередь до capacity, затем из отдельного потока вызвать `put()`. | Вызывающий поток блокируется до освобождения места.             |
| `BlockingTakeOnEmpty` | Из отдельного потока вызвать `take()` на пустой очереди.                   | Поток блокируется до появления элемента.                        |
| `PutUnblocksTake`     | Один поток ждёт в `take`, другой вставляет элемент.                        | Ждущий поток просыпается и забирает элемент.                    |
| `TakeUnblocksPut`     | Один поток ждёт в `put` на полной очереди, другой извлекает элемент.       | Ждущий поток просыпается и вставляет элемент.                   |
| `TryPutFull`          | `tryPut` на полной очереди.                                                | Возвращает `false` немедленно, без блокировки.                  |
| `TryTakeEmpty`        | `tryTake` на пустой очереди.                                               | Возвращает `std::nullopt` немедленно.                           |
| `Peek`                | `peek` на непустой очереди.                                                | Возвращает головной элемент, не удаляя его; `size` не меняется. |
| `PeekEmpty`           | `peek` на пустой очереди.                                                  | Возвращает `std::nullopt`.                                      |
| `SizeCapacity`        | Проверить `size()`, `capacity()`, `empty()`, `full()` в разных состояниях. | Значения корректны под нагрузкой.                               |

#### 2.1.2 Многопоточные тесты
| Тест             | Описание                                                                                                                                                            | Ожидаемый результат                                                 |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------- |
| `MpmcNoLoss`     | 4 продюсера, 4 консьюмера, каждый продюсер вставляет уникальные целые (сумма известна), консьюмеры суммируют полученное. Очередь с большой ёмкостью, много опреций. | Сумма вставленных равна сумме извлечённых (нет потерь, дубликатов). |
| `MpmcContention` | Продюсеры и консьюмеры работают одновременно, ёмкость маленькая (1-2), вызывая частые блокировки/пробуждения.                                                       | Отсутствие взаимоблокировок, все элементы рано или поздно извлечены |
| `MpmcFairness`   | Несколько продюсеров блокируются на полной очереди; консьюмеры начинают работать с задержкой. Проверяется, что все продюсеры в итоге вставили свои элементы         | Количество успешных `put` = общее число попыток                     |

#### 2.1.3 Тесты отмены
| Тест                           | Действия                                                                                                  | Ожидаемый результат                                                        |
| ------------------------------ | --------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| `CancelPut`                    | Заполнить очередь, запустить поток, вызывающий `put` с токеном. Основной поток вызывает `request_stop()`. | Поток просыпается, выбрасывает исключение, очередь остаётся без изменений. |
| `CancelTake`                   | Аналогично для `take` на пустой очереди.                                                                  | Исключение, элемент не извлечён.                                           |
| `CancelTimeout`                | Поток ждёт в `take(timeout)` с большим таймаутом, получает отмену до истечения.                           | Исключение, таймаут игнорируется.                                          |
| `CancelOneDoesNotAffectOthers` | Два консьюмера ждут в `take`, одному отмена.                                                              | Второй продолжает ждать, после вставки элемента забирает его.              |

#### 2.1.4 Особые ситуации

| Тест                           | Действие                                                        | Ожидаемый результат                                                                                       |
| ------------------------------ | --------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| `ConstructorZeroCapacity`      | `Queue(0)`                                                      | `std::invalid_argument`                                                                                   |
| `DestructorWithBlockedThreads` | Очередь уничтожается, пока потоки заблокированы в `put`/`take`. | Все потоки пробуждаются, деструктор завершается без падения (токен отмены передаётся владельцем очереди). |

### 2.2 Кольцевая очередь с политикой «сброс самого старого»

**Особенности:** статический буфер, продюсер никогда не блокируется, при заполнении вытесняется самый старый элемент, консьюмер может блокироваться на пустой очереди.

#### 2.2.1 Функциональные тесты
| Тест                         | Входные данные / действия                                                               | Ожидаемый результат                                                               |
| ---------------------------- | --------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| `PutNeverBlocks`             | Заполнить очередь, затем из нескольких потоков вызвать `put`.                           | Ни один продюсер не блокируется; `size` всегда равен `capacity`.                  |
| `DropOldest`                 | Очередь ёмкостью 3. Вставить `[1,2,3]`, затем `4`. Извлечь всёё.                        | Извлечены `[2,3,4]` (1 вытеснен).                                                 |
| `DropOldestSequence`         | Вставить 5 элементов в очередь ёмкостью 3, записывая историю.                           | После заполнения каждый новый сдвигает голову; порядок сохраняется без пропусков. |
| `TakeBlocking`               | `take` на пустой очереди блокируется, после `put` просыпается и забирает элемент.       | Элемент получен, соответствует вставленному.                                      |
| `TryTake`                    | `tryTake` на пустой / непустой.                                                         | Поведение согласно контракту (nullopt / значение).                                |
| `Peek` / `Size` / `Capacity` | Стандартные проверки.                                                                   | Корректная работа                                                                 |
| `FullAndEmpty`               | `full()` возвращает `true` после заполнения; `empty()` после извлечения всех элементов. | Корректная работа                                                                 |

#### 2.2.2 Многопоточные стресс‑тесты
| Тест                      | Описание                                                                                                                                             | Ожидаемый результат                                                                       |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| `MpmcNoProducerBlocking`  | 4 продюсера непрерывно вставляют, 2 консьюмера медленно извлекают (с задержкой).                                                                     | Продюсеры не блокируются, в любой момент `size ≤ capacity`.                               |
| `ConcurrentTakeWakeup`    | Несколько консьюмеров ждут на пустой очереди, продюсер вставляет один элемент.                                                                       | Только один консьюмер получает элемент, остальные продолжают ждать.                       |

#### 2.2.3 Тесты отмены
- Отмена применяется только к блокирующим `take`. Тесты аналогичны 2.1.3

#### 2.2.4 Исключительные ситуации
- `Queue(0)` → `std::invalid_argument`.

### 2.3 Кольцевая очередь с политикой «блокировка писателя» 

**Особенности:** статический буфер, продюсер блокируется при заполнении, консьюмер блокируется при опустошении. Полная симметрия.

Функциональные тесты и сценарии аналогичны **2.1** (блокирующая очередь на списке), но с учётом фиксированной ёмкости и отсутствия динамической аллокации.

| Тест                        | Описание                                                                                                                            |
| --------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| `FairnessMultipleProducers` | Три продюсера блокируются на полной очереди в порядке A,B,C. Консьюмеры извлекают три элемента. Порядок вставки должен сохраниться. |
| `FairnessMultipleConsumers` | Аналогично для консьюмеров на пустой очереди                                                                                        |
| `NoSpuriousLoss`            | Проверить что при большой нагрузке элементы не теряются                                                                             |


### 2.4 Приоритетная очередь на основе min‑heap с сохранением FIFO при равных приоритетах

**Особенности:** элементы имеют ключ приоритета. Извлечение всегда возвращает элемент с минимальным ключом; среди равных - в порядке вставки (FIFO). Блокировка продюсера при заполнении, блокировка консьюмера при опустошении.

#### 2.4.1 Функциональные тесты
| Тест               | Действия                                                                               | Ожидаемый результат                                |
| ------------------ | -------------------------------------------------------------------------------------- | -------------------------------------------------- |
| `PriorityOrder`    | Вставить элементы с приоритетами `[3,1,2]`. Извлечь все.                               | Порядок извлечения: 1, 2, 3.                       |
| `FifoAmongEqual`   | Вставить `[(prio=1, seq=0), (prio=2, seq=1), (prio=1, seq=2)]`.                        | Порядок: prio 1 seq 0, prio 1 seq 2, prio 2 seq 1. |
| `BlockingBehavior` | Аналогично блокирующей очереди `put` блокируется при заполнении, `take` - при пустоте. |                                                    |
| `Peek`             | `peek` возвращает элемент с минимальным приоритетом, не удаляя.                        | Приоритет минимален, FIFO среди равных.            |

#### 2.4.2 Многопоточные тесты
| Тест                 | Описание                                                                                                                                                                                                                       |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `MpmcPriorityStress` | Несколько продюсеров вставляют элементы со случайными приоритетами, несколько консьюмеров извлекают. Проверяется, что извлечённая последовательность глобально упорядочена по приоритету, а среди равных – по времени вставки. |
| `MpmcFairness`       | Проверка, что ожидающие продюсеры/консьюмеры обслуживаются корректно (с учётом приоритетов элементов).                                                                                                                         |
**Тесты отмены** – аналогично блокирующим очередям для `take` (и `put`, если bounded).

### 2.5 Lock‑free очередь Michael‑Scott

**Особенности:** неограниченная (или ограниченная?) обычно неограниченная, wait‑free/lock‑free, без блокировок, динамическая аллокация узлов, MPMC.

#### 2.5.1 Функциональные тесты
| Тест                   | Действия                                                                                | Ожидаемый результат                    |
| ---------------------- | --------------------------------------------------------------------------------------- | -------------------------------------- |
| `FifoOrder`            | Вставить N элементов, извлечь N.                                                        | Строгий FIFO.                          |
| `TryPutAlwaysSucceeds` | `tryPut` (или `push`) всегда успешен (очередь не ограничена).                           | Возвращает `true`.                     |
| `TryTakeEmpty`         | `tryTake` на пустой очереди.                                                            | Возвращает `nullopt` (или эквивалент). |
| `Peek`                 | Не гарантируется из‑за lock‑free, но если есть - должен возвращать голову без удаления. |                                        |

#### 2.5.2 Многопоточные стресс‑тесты
| Тест                | Описание                                                                            | Ожидаемый результат                                                                                           |
| ------------------- | ----------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| `MpmcNoLoss`        | Аналогично другим: M продюсеров, N консьюмеров, уникальные числа, большая нагрузка. | Сумма вставленных = сумме извлечённых, отсутствие дубликатов, гонок данных (TSan чист).                       |
| `HighContention`    | Много потоков одновременно `push`/`pop` на одном ядре.                              | Нет зависаний, ABA‑проблема решена (проверяется косвенно через отсутствие потерянных/повреждённых элементов). |
| `MemoryReclamation` | Длительный прогон, мониторинг памяти.                                               | Память не утекает (узлы возвращаются).                                                                        |

#### 2.5.3 Тесты отмены – отсутствуют (нет блокирующих операций).

#### 2.5.4 Исключительные ситуации
- Конструктор без параметров, деструктор корректно освобождает все узлы даже при наличии незавершённых операций.
